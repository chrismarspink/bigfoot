# Bigfoot CA — 인증·인가 도입 플랜 (PLAN ONLY)

> **상태: 미구현.** 현재 코드는 인증이 없다(`guard` 미들웨어는 통과, `actorOf` 는 `X-User` 헤더 placeholder).
> 본 문서는 인증 도입 시의 설계만 확정한다. 구현은 별도 페이즈에서 진행한다.
> 그때까지 본 콘솔/API 는 **신뢰된 망에서만** 노출할 것.

## 1. 왜 지금은 미적용인가
- P0 의 목표는 "분리 + 범용 CA 골격" 검증. 인증을 붙이기 전 기능·서명·감사 흐름을 먼저 확정.
- CA 관리 화면은 발급·폐기 권한을 쥔 **고가치 표적** → 인증은 반드시 들어오되, 설계를 먼저 굳힌다.

## 2. 인증 모델 (확정)
- **OIDC (Keycloak 또는 고객 IdP)**, Authorization Code Flow.
- **토큰 서명검증 필수**: JWKS 로 RS256 검증 + `exp`/`iss`/`azp(또는 aud)` 검증. 검증 실패 = 401 (fail-closed).
  - 참고 구현: 기존 TrustLink `admin-bff/oidc.go`(stdlib 전용 RS256, JWKS 회전 대응) 이식 가능.
- **세션**: 서버측 세션(파일/Redis). P0 무상태 원칙 유지 위해 외부 세션 권장(HA 는 보류이나 인터페이스는 교체식).

## 3. 인가 (RBAC) — 3역할
| 역할 | 권한 |
|---|---|
| `ca-auditor` | 읽기 전용: info/certs/audit/crl/root 다운로드 |
| `ca-operator` | + 발급/CSR서명/재발급/파일서명/수신자 관리 |
| `ca-admin` | + 폐기, 승인 워크플로 결재, 설정 |

- 매핑은 OIDC 그룹/롤 클레임 → 역할. 라우트별 최소권한.

## 4. 코드 통합 지점 (이미 마련됨)
- **`guard(h)`** (main.go): 현재 통과. 도입 시 여기서 (a) 토큰 검증 (b) 역할 확인 후 거부/통과.
  - 라우트별 필요한 최소 역할을 인자로 받도록 확장: `s.guard("ca-operator", s.apiCAIssue)`.
- **`actorOf(r)`** (main.go): 현재 `X-User` 헤더. 도입 시 검증된 토큰 subject 를 `guard` 가 컨텍스트/헤더로 주입 → `actorOf` 가 그 값을 읽음. **핸들러·SoR 기록 코드는 무변경.**
- **감사**: 모든 쓰기는 이미 SoR 에 actor 와 함께 기록 → 인증 도입 시 actor 가 실제 검증된 신원으로 바뀜.

## 5. 승인 워크플로(4-eyes)와의 결합 — P2-2
- 발급/폐기를 **요청(request) → 승인(approve)** 2단계로. **요청자 ≠ 승인자** 강제.
- 인증 미도입 상태에서도 P2-2 의 상태머신·API·UI 는 구현하되, 신원 분리는 `actorOf` 에 의존 →
  인증 도입 시 자동으로 "검증된 서로 다른 두 신원"이 강제된다. (그 전까지는 X-User 헤더 기반, 경고 표기.)
